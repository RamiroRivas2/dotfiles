Flows

Complete recipes, every keystroke, starting from "the computer is on and nothing is open." Each step says what you type AND what you should see, so you always know if you're on track.

Flow 0 · Open a terminal & know where you are

Every other flow starts here. Your machine has TWO worlds and the #1 beginner confusion is not knowing which one you're typing into.

  1. Press the Win key, type wezterm, press Enter. A translucent purple window opens.
  2. Look at the prompt. You are in Windows (PowerShell). It looks like:
    Windows ~ ❯  — folders look like C:\Users\ramir\...
  3. Type wsl and press Enter. Now you are in Ubuntu (Linux):
    WSL ~ ❯  — folders look like /home/ramiro/... and your projects are in ~/github
  4. To go back to Windows: type exit, Enter.
Quick identity check any time: type pwd Enter. An answer starting C:\ = Windows; starting /home or /mnt = Ubuntu. The agent tools (herdr, no-mistakes, gnhf, treehouse, firstmate) all live in Ubuntu.

Flow 1 · The daily driver (herdr + Claude + Neovim)

The workflow from the video. Do this until it's muscle memory.

first time only

  1. Flow 0 into Ubuntu, then: herdr integration install claude Enter — teaches the sidebar to read Claude's real status.
  2. Start herdr once (next list), press Ctrl+a then Shift+A, and complete the one-time Claude login in the browser it points you to.

every day

  1. WezTerm → wsl → cd ~/github/dotfiles-wsl (or whatever project).
  2. herdr Enter. You'll see a sidebar on the left and your shell on the right. (If it says it can't start, you're probably already inside herdr.)
  3. Ctrl+a then Shift+A — Claude Code opens in a pane. The sidebar row starts showing working / idle / blocked.
  4. Ctrl+a then v — a second pane opens on the right. Type nvim . Enter. That's the Kun layout: agent left, editor right.
  5. Talk to Claude on the left; look at code on the right. Click any pane to focus it (mouse works).
  6. Done for now? Ctrl+a then q — you're back in plain Ubuntu, and everything keeps running. Close the window, walk away.
  7. Hours later: WezTerm → wsl → herdr — everything is exactly where you left it.

✅ You've got it when: you can leave and come back without losing Claude's conversation.

Flow 2 · Push through no-mistakes (your question!)

Goal: make a change, and instead of pushing straight to GitHub, let the AI pipeline review → test → fix it first. We'll use your real dotfiles-wsl repo as the example.

first time only (one minute)

  1. Flow 0 into Ubuntu.
  2. no-mistakes daemon start Enter — starts the background worker. Then no-mistakes daemon status should say running.
  3. no-mistakes doctor Enter — a checklist prints. You want green on git, gh, claude, daemon. (Yours is green except the daemon you just started.)
  4. Go to the repo and add the gate: cd ~/github/dotfiles-wsl then no-mistakes init Enter. It says it added a remote called no-mistakes. That's it - one-time per repo.

every time you have a change

  1. Be in the repo: cd ~/github/dotfiles-wsl
  2. Make a branch for the change: git switch -c my-change Enter. (Branch = a safe copy of the timeline; nothing touches main until the end.)
  3. Do the work — edit files with nvim, or tell Claude what to change. Then stage and commit:
    add Enter then git commit -m "describe the change" Enter
  4. THE moment — push to the gate instead of GitHub:
    git push no-mistakes my-change Enter
    The terminal accepts it quickly; the pipeline now runs in the background.
  5. Watch it work: no-mistakes attach Enter — a full-screen dashboard opens showing steps flowing: intent → rebase → review → test → document → lint → push → pr → ci. AI reviewers read your change; if a step finds problems, it fixes them or asks you.
  6. If it pauses on a question (an "approval gate"), read what it found and pick approve / fix / skip with the keys it shows.
  7. When the pipeline finishes: it pushed the branch to GitHub and opened a pull request for you. It prints the PR link — Ctrl+click it (or check github.com/RamiroRivas2/dotfiles-wsl).
  8. On the PR page, review and click Merge when happy. Then locally: m (back to main) and pull (get the merged result).

✅ You've got it when: a PR appears on GitHub that you never manually created. ⚠️ Habit warning: plain push still goes straight to GitHub and skips ALL of this — the gate only runs when you push to no-mistakes.

If nothing seems to happen after the push: no-mistakes daemon status — the daemon being off is cause #1. Then no-mistakes runs to see history, no-mistakes attach to re-open the dashboard.

Flow 3 · Ask Claude to change code, review it, ship it

The basic agent loop — works in any repo, either world.

  1. Terminal → go to the project (cd ~/github/... in Ubuntu, or cd Documents\GitHub\... on Windows).
  2. Make a branch: git switch -c fix-thing
  3. claude Enter (or cc to skip permission prompts once you trust the repo).
  4. Describe the goal in plain English, like you talk to me. Let it work; answer its questions.
  5. When it says done, YOU verify: run the app/tests, or eyeball the changes with lazygit (arrow keys to browse files, Enter to see the diff, q to quit).
  6. Happy? add → git commit -m "what changed" → push (or git push no-mistakes fix-thing to use Flow 2's gate).
  7. Not happy? Tell Claude what's wrong and loop; or throw it all away: git checkout . (careful: discards ALL uncommitted edits).

✅ You've got it when: commits on GitHub describe work you delegated instead of typed.

Flow 4 · Overnight run (gnhf)

You sleep, Claude grinds. Run it INSIDE herdr so it survives the window closing.

before the first real night

  1. Windows: Settings → System → Power → set "Sleep when plugged in" to Never, and plug the machine in. (WSL can't keep Windows awake by itself — a sleeping laptop = frozen run.)
  2. Check your Claude credits/usage — every iteration spends real usage.
  3. Do a tiny daytime trial: gnhf --max-iterations 2 "add comments to the most confusing function" and watch what it does.

the night run

  1. Flow 0 → Ubuntu → herdr → go to the repo: cd ~/github/<project>
  2. Clean tree check: git status must say "nothing to commit". Commit or stash anything pending — gnhf refuses a dirty tree.
  3. Launch with caps: gnhf --max-iterations 25 --stop-when "all tests pass" "your objective here"
    A starfield dashboard appears; it creates a branch like gnhf/your-objective and starts iterating.
  4. Ctrl+a q to detach. Close everything. Sleep.
  5. Morning: wsl → herdr → see where it got. Read the diary: cat .gnhf/runs/*/notes.md
  6. Review the commits: git log --oneline main..gnhf/your-objective and the full change: git diff main...gnhf/your-objective
  7. Keep it: m then git merge gnhf/your-objective. Or keep parts (git cherry-pick <commit>). Or delete the branch and lose nothing: git branch -D gnhf/your-objective

✅ You've got it when: you wake up to commits worth keeping. If it stopped early, three failures in a row or empty credits are the usual reasons — notes.md tells the story.

Flow 5 · Change your own setup (example: add an alias)

Say you want st to mean git status. Same pattern for any config change.

Windows side

  1. cd Documents\GitHub\dotfiles
  2. nvim home\powershell\profile.ps1
  3. Find the aliases, add a line: function st { git status } (press i to type, Esc when done - it saves)
  4. :q to leave · open a NEW terminal · type st - works
  5. add → git commit -m "add st alias" → push

WSL side

  1. cd ~/github/dotfiles-wsl
  2. nvim home.nix - find shellAliases, add: st = "git status";
  3. Esc then :q
  4. ./rebuild.sh - Nix applies it · open a new shell · st works
  5. add → git commit -m "add st alias" → push

Same loop for the prompt (starship), terminal look (wezterm.lua - saves apply instantly), Neovim, herdr keys (then Ctrl+a Shift+R to reload), and agent rules (AGENTS.md - on Windows run .\rebuild.ps1 after).

Flow 6 · First voyage with firstmate (when you're ready)

  1. Flow 0 → Ubuntu. Start tmux (firstmate's engine - not herdr for this one): tmux new -s firstmate
  2. cd ~/github/firstmate then claude - that's the whole launch. It introduces itself as your first mate and checks its tools.
  3. It will ask to install one missing helper (tasks-axi) - say yes.
  4. Give it a safe, read-only first job:
    "ahoy! scout my github project RamiroRivas2/InventorySystem - report on how invoices are generated"
  5. It clones the repo under projects/ and spawns a crew window named fm-.... Peek at it: Ctrl+a w, pick the fm window, look but don't type (typing there = giving orders). Ctrl+a w back to window 0.
  6. Wait - the report lands and firstmate summarizes it in your chat. For real changes later, it hands you a PR link and waits for your "merge it".
  7. Detach with Ctrl+a d; return with tmux attach -t firstmate.

⚠️ Every crewmate is a full Claude session - 3 parallel tasks ≈ 3x usage. Start with one.

The universal ending of every flow: add → git commit -m "..." → push. If you did that, your work is safe on GitHub and no mistake is permanent.
