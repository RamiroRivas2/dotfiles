4 · herdr WSL - your multiplexer

Keeps shells and agents running in a background server - detach, close everything, come back later and it's all still alive. The sidebar shows each agent's state (working / blocked / done / idle). Mouse works everywhere: click panes, drag borders, right-click for menus, drag-select to copy, Ctrl+click links.

Forgot a key? Ctrl+a ? shows the live built-in cheat sheet. Docs online say Ctrl+b - yours is remapped to Ctrl+a.

Your first session, step by step
  1. WezTerm → wsl → cd ~/github/dotfiles-wsl (any project)
  2. herdr - first run starts the server; sidebar left, shell right
  3. One-time: herdr integration install claude
  4. Ctrl+a Shift+A - Claude opens; first ever run asks you to log in (browser)
  5. Ctrl+a v - shell pane beside it; run nvim . there for the Kun layout
  6. Detach with Ctrl+a q, close the window, walk away
  7. Later: wsl → herdr - everything is exactly where you left it
Traps to know
  • Ctrl+a q detaches (keeps running); herdr server stop KILLS everything including Claude. Don't confuse them.
  • After a server restart your layout comes back but running programs don't - shells return empty (Claude can resume itself if the integration is installed).
  • Ctrl+a normally means "jump to line start" in shells - herdr eats it as the prefix. Use Home instead.
  • herdr inside herdr is blocked - if it refuses to start, you're already in it.
  • New tabs pause to ask for a name - it's a prompt, not a hang.
  • herdr config reset-keys would wipe your Ctrl+a prefix and the Shift+A Claude binding.

Gray text on lines = git blame (who, when) - automatic via gitsigns. :Lazy opens the plugin manager (U updates plugins, q closes).

oil - the file browser that's a text file
  1. Space e - the folder appears as an editable list
  2. Enter opens a file/folder · - goes up one level
  3. RENAME: edit the filename like any text (i, type, Esc)
  4. DELETE: dd the line · NEW FILE: add a line with the name (end folders with /)
  5. :w - a confirmation shows exactly what will happen; accept and it's done for real
neogit - commit and push without leaving the editor
  1. Space g - status view: changed files listed
  2. s on a file = stage it (u unstages) · Tab peeks at the diff
  3. cc - commit editor opens; write the message, then :wq
  4. P then p - push · q - back to your code

7 · The workflow tools

skills CLI both

Package manager for agent skills - "npm install, but for Claude abilities". Browse the registry at skills.sh.

npx -y skills findbrowse the registry interactively (or find react to search)
npx -y skills add owner/repo -lPREVIEW what a repo contains - installs nothing
npx -y skills add owner/repo --skill name -g --copyinstall one skill, user-level; --copy always on Windows (OneDrive + symlinks don't mix)
npx -y skills ls / ls -glist project / global skills (scopes are separate!)
npx -y skills update -yrefresh installed skills (run every few weeks)
npx -y skills remove nameuninstall
npx -y skills use repo@skill | claudetry a skill ONCE without installing
npx -y skills init my-skillscaffold your own skill

Traps: installs default to PROJECT scope (use -g for everywhere) · Windows and WSL are separate installs · new skills only appear in NEW Claude sessions · skills are instructions your agent will follow - only install from repos you trust.

lavish you're inside it now

Claude writes an HTML page, it opens in your browser, you click/select anything and attach notes, Claude reads them and revises. You rarely type these commands - the skill runs them - but they exist:

(you, in chat) "...as a lavish artifact"the trigger phrase - Claude does the rest
(you, in browser) click / select textattach a note; queue as many as you want; hit send
npx -y lavish-axi file.htmlopen/resume a review session
npx -y lavish-axi export file.html --out copy.htmlportable single-file copy (email-able)
npx -y lavish-axi share file.html --password pwpublish to ht-ml.app - PUBLIC by default, use a password for anything private
npx -y lavish-axi end file.html / stopend a session / stop the server (the reset button)

Never share artifacts with business data (Stockroom invoices!) without --password. A session you ended in the browser politely refuses to reopen - that's by design.

no-mistakes use from WSL

The quality gate: you push to a local "no-mistakes" remote instead of GitHub; a daemon runs a pipeline (intent → rebase → review → test → document → lint → push → pr → ci) and only clean code lands.

no-mistakes doctorhealth check: git, gh, daemon, which agents it found
no-mistakes daemon start / statusstart / check the background service (currently stopped on your machine - start it before first use)
no-mistakes initonce per repo (repo needs an origin remote): adds the gate
git push no-mistakes <branch>THE new habit - push here instead of origin
no-mistakes attachfull-screen TUI to watch the pipeline and answer approval gates
no-mistakes runs / status / statshistory / current repo state / usage stats
no-mistakes --skip test,lintskip steps for one run
no-mistakes rerunre-run after a failed/cancelled outcome (never mid-run)
no-mistakes ejectremove the gate from a repo
no-mistakes axi run --intent "goal"agent mode - tell Claude to use this to gate its own work

Traps: plain git push origin still bypasses the gate · daemon off = pushes sit unprocessed · each run drives Claude and spends real usage · use it on repos living in ~/github (WSL), not /mnt/c.

gnhf caps on, branch first

"Good night, have fun" - one objective, Claude loops on it unattended; every success is a commit, every failure rolls back, notes.md carries memory between rounds.

gnhf --max-iterations 2 "small task"your first trial run - ALWAYS set a cap until you trust it
gnhf "objective"full run - creates a gnhf/<slug> branch, commits each iteration
gnhf (no prompt, on a gnhf/ branch)resume where it left off
cat prd.md | gnhf --max-iterations 15pipe a long spec as the objective
--stop-when "all tests pass"natural finish line instead of a count
--max-tokens 5000000hard budget cap
--worktreerun in a separate worktree (several runs in parallel)
--pushpush the branch after every successful iteration
Ctrl+C once / twicegraceful stop (lets iteration finish) / force stop

Review after: git log --oneline main..gnhf/branch and git diff main...gnhf/branch; the story is in .gnhf/runs/<id>/notes.md. Traps: needs a CLEAN tree to start · overnight WSL runs need Windows power settings set to not sleep · check your Claude credits before bed · run inside herdr/tmux so it survives the window closing.

gh-axi for Claude, mostly

GitHub CLI rebuilt for agents - compact output, fewer mistakes, cheaper tokens. Say "use gh-axi" when Claude does GitHub work. Your part is just verifying it works:

gh-axidashboard: detected repo + open issue/PR counts
gh-axi issue list --state openreal test - compact typed list proves wrapper + gh + auth all work
gh-axi pr view 42 --reviews / pr checks 42 / pr diff 42inspect PRs (add --full for untruncated)
gh-axi issue list -R owner/nametarget another repo (-R goes AFTER the command)
gh-axi update --checkversion check - Windows and WSL copies update separately
gh-axi setup hooksoptional: Claude sessions start with ambient repo context

Traps: write commands (merge, close, delete) hit GitHub immediately, no extra confirmation · needs gh installed AND logged in · terse output is by design, it's for agents.

treehouse WSL

Pool of ready-to-use repo copies (git worktrees) so parallel agents don't collide. firstmate drives it automatically; by hand:

treehouse gethands you a fresh worktree in a subshell; exit returns it. First thing inside: git switch -c my-branch (it starts detached!)
treehouse statuswhat's out, what's free (run inside the repo)
WT=$(treehouse get --lease)script/agent mode: reserved path, no subshell; release with treehouse return "$WT"
treehouse prune → prune --yesdry-run then actually reclaim disk from stale worktrees
treehouse destroy path --yesremove one specific worktree (dry-run first without --yes)

Traps: commit + push BEFORE exiting a worktree - returned ones get recycled · --include-unlanded = irreversible data loss · pool lives at ~/.treehouse in WSL.

firstmate the deep end WSL

Not a program - a repo whose AGENTS.md turns Claude into a captain's first mate that spawns crew agents in tmux windows, each in its own worktree, and brings you finished PRs. You talk to ONE agent in plain English.

tmux new -s firstmate then cd ~/github/firstmate && claudethe entire launch - AGENTS.md takes over
"ahoy! scout my github project RamiroRivas2/X - report on ..."a safe read-only first task (reports land at data/<id>/report.md)
"fix the flaky login test and add dark mode"ship tasks - it clones the repo under projects/, spawns crew, hands you a PR
"merge it"your approval - nothing merges without it
Ctrl+a whop between your chat window and crew windows (LOOK, don't type - typing in a crew window is a captain's order)
/afk · /stow · /updatefirstmatecheap away-mode supervision · save session knowledge · self-update

First run will ask to install tasks-axi - say yes. Traps: each crewmate is a full Claude session (3 tasks ≈ 3x usage) · give it GitHub repo names, never /mnt/c paths · put standing preferences in data/captain.md · leave +yolo off until you trust it.

Dictation: Win+H anywhere in Windows types what you say - works pointed at a herdr pane running Claude. If you love it, ask me to install Handy (open-source, offline).
